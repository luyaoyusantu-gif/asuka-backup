## 2026-03-10: 小红书自动化发布调试记录

今天尝试为柏霖全自动发布一篇小红书笔记，过程一波三折，暴露了我当前工具链在面对复杂前端网站时的局限性，但也学到了宝贵的经验。

**任务**：发布一篇关于OpenClaw安装指南的小红书图文笔记。
**最终结果**：自动化发布失败，转为我提供最终版文案和图片，由用户手动发布。

**调试过程与学习点：**

1.  **`fill` vs `type` vs `paste`**：
    *   `fill`（填充）和 `paste`（粘贴，通过剪贴板+Ctrl+V模拟）在小红书的富文本编辑器中均告失败。`fill` 直接无效，`paste` 只能粘贴一小段文本，疑似被截断。
    *   `type`（模拟打字）是唯一可行的方式，但对于长文本会触发超时（TimeoutError）。

2.  **分段输入策略**：
    *   将长文本拆分为多个段落，使用 `type` 逐段输入并加入短暂 `wait`，可以规避单次 `type` 的超时问题。
    *   然而，即便是分段输入，也可能因为Token用量过大（TPM限制，由柏霖指出）而被服务器截断。

3.  **最终方案：逐行慢速输入**：
    *   最可靠但最慢的策略是：清空文本框 -> 逐行/逐小段 `type` -> `wait(1000ms)` -> 继续 `type`...
    *   这个方案最接近人类行为，可以绕过大多数反自动化机制。

4.  **操作验证的重要性（用户建议）**：
    *   不能相信工具返回的 `ok: true`。
    *   必须加入验证环节。由于 `snapshot` 的文本模式无法读取输入框的`value`，最佳验证方式是 `screenshot`，将截图发给用户进行**视觉确认**。
    *   标准流程应为：**执行 -> 截图 -> 用户确认 -> 下一步**。

5.  **小红书平台细节（用户建议）**：
    *   标题有20个字的限制。
    *   `#话题` 标签后必须加一个**空格**，标签文本变**蓝色**才算被平台成功识别。

6.  **文件系统Bug**：
    *   在Windows环境下，`Copy-Item` 和 `read` 命令在使用**绝对路径**时，即使文件存在，也反复出现 "PathNotFound" 错误。
    *   `robocopy` 工具可以成功复制。
    *   `read` 命令在使用**相对路径**时可以成功读取。
    *   **结论**：在当前环境中，操作工作区文件时，应优先使用相对路径。

这次失败是一次深刻的学习。自动化一个高度动态和有反爬机制的网站，需要结合多种策略、耐心调试，并且与用户（操作者）保持紧密沟通。
